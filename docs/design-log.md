# Design Log — Toss PD 과제: "6명이 회의 일정을 잡는 경험"

> 이 파일은 우리의 설계 토의를 **누적 기록**하는 단일 소스입니다.
> 규칙: 솔루션을 먼저 제안하지 않는다. 문제 분해 → 가설 → 근거 → 약점 → 다음 리서치 순서로만 진행한다.
> 모든 주장에는 근거를 요구하고, 근거 없는 주장은 반박한다.
> 새 토의가 끝날 때마다 이 파일 맨 아래에 항목을 **추가(append)**한다. 기존 항목은 지우지 않는다.

## 기록 포맷 (매 항목 이 구조 유지)
- **Fact** — 현재까지 확인된 사실
- **Hypothesis** — 우리가 세운 가설
- **Evidence** — 근거 (논문 / UX 리서치 / Reddit / 실제 경험 / 기존 서비스 분석, 출처 URL 포함)
- **Weakness** — 현재 논리의 빈틈
- **Next Research** — 다음으로 조사할 질문

## Problem Frame (합의된 전제)
- 과제: "6명이 회의 일정을 잡는 경험을 설계하라."
- 조건: 같은 회사 동료 6명 / 다음 주까지 / 1시간 / 선호 상이 / 점심 직후 회피자 존재 / 특정 요일 외근자 존재 / 필수·선택 참석자 구분.
- 사용자의 초기 프레이밍: "이건 Calendar App이 아니라 Meeting Decision Service다." → **아직 미검증 가설로 취급** (병목이 '정보 수집'인지 '결정'인지 미확정).

## 미해결 쟁점 (Open Questions)
1. 병목이 **가용성 취합(collection)**인가 **결정(decision)**인가?
2. pain의 주체가 **조직자**인가 **참석자 5명**인가? (노동 비대칭)
3. 조율을 어렵게 만드는 것이 **선호(preference)**인가 **가용성(availability)**인가?
4. "필수/선택" 조건 → 완벽한 슬롯이 없을 때 **누가, 어떻게 양보를 납득하는가**.

## Entry 001 — 초기 프레이밍 반박 (2026-07-02)
- **Fact**: 사용자가 문제를 "Meeting Decision Service"로 프레이밍함. 근거는 아직 없음.
- **Hypothesis**: (검증 대상) 병목은 결정이 아니라 가용성 취합에 있을 수 있다.
- **Evidence**: 미수집 (Entry 002에서 리서치 착수).
- **Weakness**: 사용자 프레이밍이 결론을 선점함. 병목 위치가 미확정인 채로 서비스 성격을 단정.
- **Next Research**: 실제 사용자 불만 + HCI 논문 + 기존 서비스 한계로 병목 위치를 특정.

## Entry 002 — User Journey & 병목 위치 리서치 (2026-07-02)
- **Fact (근거 기반)**:
  - 그룹 회의 스케줄링은 8단계 여정으로 분해됨: ①need ②required/optional 선정 ③후보 시간 제안 ④**가용성 취합** ⑤충돌·선호 조정 ⑥최종 슬롯 결정 ⑦통보·확정 ⑧변경/노쇼/리스케줄.
  - **가장 흔한 stall 지점은 ④가용성 취합** — 사람들이 응답을 안 함. "back and forth communication"이 최대 시간 킬러이며 참여자 수에 따라 증가 (ETH Zurich/Doodle 연구, n=96).
  - 5명 초과 시 공통 시간 존재 확률이 **상전이(phase transition)처럼 급락** (Case Western, EPJ B, 2024) → ⑤/⑥에서 "가능한 슬롯 자체가 없음"이라는 수학적 실패 모드 존재.
  - **노동 비대칭은 문서화된 불만**: 부킹링크(Calendly)=선택 노동을 상대에게 전가("temporal autonomy 포기" 비판, Westenberg/Medium). 폴/그리드(Doodle·When2meet)=참석자 N명에게 반복 수기 입력·자기기억 의존 강요 → ④ 비응답의 주원인.
- **Hypothesis**: 진짜 unmet need = "참석자 노동을 0에 가깝게(실제 캘린더 자동 반영, 그리드/링크 없이) 하면서 조직자는 필수/선택 결정권을 유지" → ④ stall 붕괴.
- **Evidence (URL)**:
  - Doodle/ETH study: https://doodle.com/en/resources/blog/study-reveals-time-spent-with-scheduling/
  - SciTechDaily(EPJ B phase transition): https://scitechdaily.com/a-mathematical-nightmare-new-research-explains-why-scheduling-meetings-is-so-hard/
  - Calendly "is it rude": https://calendly.com/blog/is-calendly-rude
  - Westenberg(Medium): https://medium.com/westenberg/why-i-hate-booking-links-904f5e26d8f8
  - meetergo(When2meet 한계): https://meetergo.com/en/magazine/when2meet
- **Weakness**:
  - "43% spend 3+ hrs/week scheduling" = 1차 출처 불명, directional로만 사용.
  - Doodle 연구는 벤더 발행 학사논문(n=96) → 독립적 아님.
  - 위 근거는 대부분 서구/영어권 맥락. **국내 실무(슬랙/카톡 기반 "다들 언제 돼요?")** 맥락 미검증.
- **Next Research**: (진행 중) ①실제 사용자 불만 원문(Reddit/Quora/리뷰) ②HCI/학술 근거(availability vs preference, 비응답 심리). 완료 시 Entry 003·004로 추가.

## Entry 003 — 기존 서비스 실사용자 불만 (2026-07-02)
- **Fact (근거 기반)**:
  - **공통 병목 = back-and-forth**: 회의 1건 예약에 이메일 약 7.3통, 그룹은 30통/30분+로 팽창. 모든 툴이 이걸 풀겠다고 주장.
  - **그룹(N:N)이 미해결 케이스**: Calendly·Google "Find a time"·FindTime은 1:1 또는 단일 조직자 흐름에 최적화. 진짜 "N명 공통 시간 찾기"는 결국 폴링/수동 추적으로 붕괴.
  - **루프를 안 닫음**: When2meet(알림 전무), Doodle(비응답자 미통지), FindTime 모두 조직자가 수동으로 응답 추적·확정.
  - **조직 간/시간대에서 깨짐**: Google Find-a-time은 캘린더 공유 필요 → 외부 참석자 빈칸. When2meet/Doodle 시간대 처리 혼란·오류.
  - **그리드/다단계 투표가 마찰**: Doodle 재설계 그리드·2단계 투표, When2meet 드래그 그리드(모바일 최악)가 "이게 이렇게 어려울 일인가" 유발.
  - **Calendly 고유의 사회적 비용**: 링크 보내기가 "get in line" 권력과시로 인식(Lessin/Fortune). 단, 논쟁적이며 오히려 성장 촉발.
- **Hypothesis**: 국내 실무는 슬랙/카톡 "다들 언제 돼요?"가 지배적일 가능성 → 위 서구 툴 불만과 다른 층위. (미검증)
- **Evidence (URL)**:
  - Calendly "rude" 논쟁: https://fortune.com/2022/02/03/calendly-outlook-social-capital-workplace-dynamics/ , https://x.com/lessin/status/1486477359717187589
  - When2meet 한계: https://meetergo.com/en/magazine/when2meet , https://wellpin.io/blog/when2meet-pros-and-cons-complete-review-of-the-free-scheduling-tool
  - Google Find-a-time 외부참석자: https://www.usecarly.com/blog/how-to-find-a-time-in-google-calendar/
  - Doodle FAQ(비응답자 미통지): https://help.doodle.com/en/articles/10295132-group-poll-troubleshooting-and-faq
  - 백앤포스 통계: https://www.usecarly.com/blog/schedule-meetings-without-back-and-forth/
- **Weakness**: **Reddit/Quora 원문 직접 접근 실패**(크롤러 차단) → 불만은 대부분 2차 리뷰/뉴스 경유. Trustpilot/CNBC는 403으로 검색 스니펫만. **불만 표본 편향**(불만자만 글 씀). 국내 맥락 여전히 미검증.
- **Next Research**: 국내 사용자(슬랙/카톡 조율) 실제 행동 확인. Reddit 원문은 사용자가 직접 인용해주거나 대체 경로 필요.

## Entry 004 — HCI/학술 근거 (2026-07-02)
- **Fact (근거 기반, Tier1 peer-reviewed)**:
  - **노동 vs 수혜 비대칭은 30년 된 정설**: Grudin(CACM 1994) — 수혜자는 조직자지만, 작동하려면 전원이 개인 캘린더를 정확히 유지해야 함. IC가 안 하므로 기능이 안 쓰임. → 우리의 "참석자 노동 0" 가설의 이론적 뿌리.
  - **스케줄링 = 가장 번거로운 지식노동 태스크**: Microsoft Research Calendar.help(CHI 2017) — 수백 명 조사에서 "회의 잡기"가 가장 번거로운 일로 지목.
  - **스케줄링은 사회적 의사결정**: Reinecke et al.(CSCW 2013) — Doodle 150만 폴 분석, 집단주의 문화가 더 일찍 응답·합의. 물류가 아니라 사회적 행동.
  - **가용성은 확률적으로 예측 가능**: Horvitz "Coordinates"(UAI 2002). 무완벽슬롯 시 비용/효용 트레이드오프는 Horvitz "Mixed-Initiative"(CHI 1999).
  - **소프트 제약(선호)이 문제를 최적화 문제로 격상**: 인사/교대 스케줄링 공정성 문헌 — 상충 선호는 타협 필요, 페널티 균등 분배가 더 공정. (인접 도메인, 전이 주의)
  - **비응답 = 이메일 과부하 결과**: Kooti et al.(WWW 2015) — 과부하 시 더 적게/짧게 응답.
- **Hypothesis (강화됨)**: unmet need = "참석자 노동을 실제 캘린더에서 자동 충당, 조직자는 필수/선택 결정권 유지, 무완벽슬롯 시 **누가 양보하나를 공정하게 배분**". Grudin+Horvitz+공정성 문헌이 뒷받침.
- **Evidence (URL)**:
  - Grudin CACM 1994: https://webpages.charlotte.edu/richter/classes/2006/6010/readings/cacm94.html
  - Calendar.help CHI 2017: https://www.microsoft.com/en-us/research/publication/calendar-help-designing-workflow-based-scheduling-agent-humans-loop/
  - Reinecke Doodle CSCW 2013: https://iis.seas.harvard.edu/papers/2013/reinecke13doodle.pdf
  - Horvitz Mixed-Initiative CHI 1999: http://erichorvitz.com/chi99horvitz.pdf
  - Kooti WWW 2015(요약): https://phys.org/news/2015-10-factors-affect-email-response.html
- **Weakness**: 공정성 근거는 **교대근무/컨퍼런스 도메인** → 6인 회의로 직접 전이 미검증. "선호를 밝히는 것의 사회적/심리적 비용"을 **인간 대상으로 측정한 논문 없음**(제약이론 privacy 모델이 최선의 대용). Google Research 등가 논문 없음. 여러 PDF는 바이너리/403 → 메타데이터 기반, 인용 전 원문 대조 필요.
- **Next Research**: (1)"선호 노출의 사회적 비용" 국내/실증 근거 보강. (2)Pain Point를 **빈도×심각도**로 우선순위화(다음 단계). (3)국내 슬랙/카톡 조율 맥락.

## Entry 005 — 과제 원문 재인식 & 프레이밍 교정 (2026-07-02)
- **Fact (과제 원문 확정)**: 캡처 원문 기준 핵심 문장 3개 —
  1. 명시된 pain: "여러 사람의 조건을 **한 번에 이해**하고 모두에게 **괜찮은 시간을 찾기** 쉽지 않다."
  2. 명시된 사용자·목표: "**사용자가** 동료들의 상황을 **파악**하고, 모두가 **괜찮은** 시간을 찾아 회의 시간을 잡는다."
  3. 조건: 같은 회사 동료 6명 / 다음 주까지(마감) / 1시간 / 점심직후 회피(소프트 선호) / 특정요일 외근(반복 제약) / 필수·선택 참석자.
- **교정 포인트 (우리 기록이 원문에서 이탈했던 부분)**:
  - 과제는 **단수 '사용자'(=조직자) 중심**이다. 나머지 5명은 원문상 "동료들의 상황(파악 대상)"이며, **이들이 제품 사용자라는 보장이 없다.** → 우리가 밀던 "참석자 노동 0"이 히어로가 아닐 수 있음. (Entry 002·004 재검토 필요)
  - 과제가 지목한 병목은 **"한 번에 이해(comprehension/legibility) + 괜찮은 시간 찾기"** 인데, 우리 리서치(Entry 002)는 병목을 **"수집/비응답"**으로 결론냈다. **두 단계가 다르다.** 어느 쪽을 진짜 문제로 볼지 미해결.
  - "같은 회사 동료" = 사내 전용 → 조직 간/외부참석자 깨짐(Entry 003의 큰 축)은 **범위 밖**. 사내 공유 캘린더/디렉터리 존재 가정 가능. 단, Grudin 함정(전원 캘린더 정확 유지)은 국내 맥락에서 여전히 미검증.
  - "**괜찮은**"(≠완벽한) + 필수/선택 + 마감 → 완벽 슬롯 최적화가 아니라 **마감 압박 하의 satisficing/양보 배분** 문제로 읽힌다.
- **Hypothesis (신규, debate 대상)**: 진짜 core는 "결정 자동화"도 "수집"도 아니라, **6명의 뒤섞인 하드(가용성)+소프트(선호) 제약을 조직자가 '한 번에' 읽을 수 있게 만드는 legibility 문제**일 수 있다. 근거: 원문 "한 번에 이해" 직접 인용.
- **Evidence**: 과제 원문(1차 소스). 기존 Entry 002~004는 방향 근거로 유지하되 프레이밍 재해석 필요.
- **Weakness**: Toss가 명시한 pain을 그대로 믿는 것도 위험(과제 설명은 힌트일 뿐 검증 대상). 반대로 우리 리서치만 믿고 원문 강조점을 무시하는 것도 위험. 둘을 화해시켜야 함. 국내 맥락 여전히 0.
- **Next Research / Decision**: (1)병목을 '수집'으로 볼지 '이해/legibility'로 볼지 결정. (2)5명이 사용자인가 데이터인가 결정 → 이게 전체 UX를 가름. (3)그 후 Q1 재작성(현재 Q1은 attendee-centric으로 치우침 → 교정 예정).

## Entry 006 — 방향 확정 & 불가능 삼각형 (2026-07-02)
- **Decision (잠금)**:
  1. **5명도 사용자다.** 단 역할은 "수집 노동"이 아니라 "선호 표현"으로 최소화. (Entry 005 화해안 채택)
  2. **캘린더 성실 사용에 의존하면 안 된다.** (사용자 명시 제약) → Google식 자동조회 배제.
  3. 해법은 흔치 않은/근본적 UX여야 하되, **참신함은 문제 재정의에서 파생**되어야 함(참신함 자체가 목표 아님).
- **Core tension (불가능 삼각형)**: [정확성] · [캘린더 비의존] · [저노동] 셋 동시 충족 불가. 기존 도구는 각각 한 꼭짓점 포기(Google=캘린더의존, Doodle/When2meet=고노동, 카톡=부정확). 우리가 노리는 빈 꼭짓점 = **비의존+저노동 → 정확성을 무엇으로 메꾸나**가 설계 급소.
- **Hypothesis (유력, 확정 전)**: 정확성 공백을 **"반복 규칙"**으로 메꾼다. 즉 매 회의 가용성 수집(one-off)을 버리고, 각자의 **고정 선호·패턴을 1회 수집→영구 재사용** + **조직자 제안에 원탭 반응/거부** + **무응답=기본값+마감**.
- **Evidence**: 원문 제약 예시가 전부 **반복형**("점심직후 회피", "특정요일 외근 많음")이고 one-off("3시에 다른 회의")가 아님 → 중요 제약은 고정 규칙이라는 원문 힌트. + Grudin(전원 입력 전제의 실패).
- **Weakness**: 반복 규칙은 **일회성 충돌**(이번주 치과)을 못 잡음 → 순수 규칙만으론 부정확, 규칙+제안거부 **하이브리드** 필요. 국내 캘린더 사용률/조율 행동 근거 여전히 0. "괜찮은 시간"의 수용 기준(필수 전원? 선택 몇 명?)도 미정.
- **Next**: 메커니즘 축 택1(반복규칙+반응 vs 순수 제안-반응). 국내 맥락 검증 여부 결정. 그 후 Q2/Q3 작성.

## Entry 007 — THE 문제 확정 & '완벽 정보' 오류 교정 (2026-07-02)
- **Decision (잠금)**: THE 문제 = **P3(뒤섞인 제약 legibility) + P4(하드/소프트 구분) 토대 → P5(양보 공정 배분) 귀결.** (P1/P2 수집은 히어로 아님, 재프레이밍으로 녹일 전제조건)
  - 한 문장: "6명의 하드 가용성+소프트 선호+필수/선택 제약을 한눈에 이해하지 못해, 마감 안에 '괜찮은' 합의 시간을 빠르고 공정하게 확정하지 못한다."
- **교정 (사용자 가정 반박)**: 사용자는 "개개인 일상을 완벽히 알면 모두 원하는 일정을 맞출 수 있다"고 가정 → **틀림.** 참여자 5명 초과 시 완벽 정보로도 공통 슬롯 존재 확률이 상전이처럼 급락(Entry 002, EPJ B 2024). 즉 문제의 본질은 **정보 부족이 아니라 양보의 수학적 불가피성.** → 기준점은 "모두 만족"이 아니라 "**양보를 공정하게 배분하는 규칙**"이어야 함.
- **Open — 기준점(수용 기준) 필요**: '괜찮은 시간'을 판정 가능하게 만들 정의 미확정. 제안(다음 debate 대상):
  - Floor(하드): 필수 참석자 전원 가능 = 후보 자격.
  - 랭킹: ①소프트 위반 총량 최소 ②위반의 공정 분산(한 사람 몰림 penalty) ③선택 참석자 수 최대 ④동점 시 마감까지 최단.
  - 미정: 기준을 조직자가 정하나/시스템 기본값인가, 공정성을 참여자에게 보이게 할 것인가.
- **원칙(파생)**: 완벽 정보 불가 → 각 사람은 **최소한이되 결정에 가장 영향 큰 신호**만 낸다(하드=반복 블록, 소프트=선호; 필수/선택은 조직자가 지정). "각자 최선의 UX" = 최소·재사용·결정관련 신호.
- **Evidence**: 과제 원문(P3/P4 지목) + Entry 002/004(상전이·Grudin). **Weakness**: 우선순위 점수는 대부분 판단(국내 정량 0). 기준점 가중치 근거 없음.
- **Next**: 기준점(수용 기준) 확정 → 그 다음 메커니즘 축 택1 → Q2/Q3.

## Entry 008 — 기준점(수용 기준) v1 확정 (2026-07-02)
- **Decision (잠금) — "괜찮은 시간" 판정 규칙 v1**:
  1. **Floor**: 필수 참석자 전원 가능 = 후보 자격.
  2. **Floor-empty 처리**: 필수 전원 슬롯이 0개면 실패 표시 대신 **하드 양보 옵션 제시** — "누가 무엇을(외근/기존일정) 조정하면 어떤 슬롯이 열리는지" 명시. (P5 양보가 하드 레벨에서도 작동)
  3. **랭킹(Floor 통과분)**: ① 소프트 위반 총량 최소 → ② **회의 내 minimax**(양보가 한 사람에게 몰리면 penalty, 이력 불요) → ③ 선택 참석자 수 최대 → ④ 마감까지 최단.

- **정정 기록**: 이전(Entry 007)에 "공정분산=이력필요→v2" 라 했으나 **회의 내 minimax 공정은 이력 불요 → v1 채택**으로 정정.
- **Next**: 메커니즘 설계(신호 수집 / 자동 계산 / legible 제시)의 **상호작용 스파인** 결정부터.

## Entry 009 — 상호작용 스파인 B & 등록 모델 (2026-07-02)
- **Decision (잠금)**: Spine **B (상시 규칙 → 자동 후보)** 채택. 각자 반복 하드 블록+소프트 선호를 1회 등록·재사용 → 시스템이 기준점으로 랭킹 후보 자동 생성 → 조직자는 선택, 참여자는 one-off 보정만.
- **Decision — 등록 처리 (gate 아님)**: 등록은 진행 조건이 아니라 **정밀도 향상(non-blocking)**. 시스템은 항상 현재 정보만으로 랭킹 후보를 낸다. → P1(수집 stall) 재발 방지. ("N명 등록 필요" 식 gate UX 금지)
  - **정정**: 사용자 초안 "나머지 5명 등록 필요/완료돼야 좁혀짐"은 gate라 P1 부활 → **narrowing(정밀도↑)은 OK, gating(완료강제)은 금지**로 교정.
  - **타겟 넛지**: 필수 미등록=Floor 신뢰 불가 → 강하게 넛지 & 후보에 "○○ 규칙 미반영" 표시. 선택 미등록=랭킹 영향만 → 약하게.
  - **마감 기본값**: 미등록자는 마감 시 기본값 처리(무한 대기 차단). 넛지는 시스템 자동(조직자 수동 추적 제거).
- **UX Writing 원칙**: "지금도 화 14시로 잡을 수 있어요, 규칙 반영되면 더 정확" (진행가능+보너스) > "3명 더 등록 필요" (대기+남탓).

- **Verdict**:

## Entry 035 — 정적 프로토타입 → Next.js+React+Node 실앱 재구성 · 2명 이상 · 공유 백엔드 (2026-07-03)

- **Hypothesis (미해결·중요)**: **Vercel 서버리스는 요청마다 인스턴스가 달라 인메모리 Map이 조직자↔참여자 기기 간 동기화를 보장 못 함**. 단일 warm 인스턴스 내 짧은 세션은 우연히 되지만 신뢰 불가. → 진짜 크로스디바이스 동기화가 필요하면 `store.ts`만 Upstash Redis/Vercel KV로 교체(1파일). 지금은 미교체 상태로 배포.
- **Evidence**: 로컬 `next build` 타입체크 통과 + `next start`에서 API E2E 검증 — 1명 생성 거부, 2명 생성·추천(화 10시로 최적: 김 월요일 불가+9시 기피, 이 점심직후 기피 회피), 10명 생성·seed·decide 정상, 홈 렌더 확인. (브라우저 확장 미연결로 클릭 스모크는 생략 — 클라 컴포넌트는 빌드 타입체크로 커버.)

## Entry 037 — 실사용 피벗: 추상 선호 토글 → 실제 날짜 범위(주간/월간) + "안 되는 시간" 그리드 (2026-07-06)
- **Fact (재진단)**: 사용자가 "너무 그 문제(양보 배분·익명)에만 집중돼 실사용이 안 된다"고 지적. 코드로 확인: 시간 세계가 `engine.makeSlots()`에 **월~금 9~17시·날짜 없음·이번 주 고정**으로 하드코딩. 참여자 입력은 실제 일정이 아니라 추상 토글(점심직후/이른아침 회피)+요일 통째 차단(`hardDays`)뿐. → **"다음 주/이번 달 안에 잡자"(범위 선택)도, "화 14~16시 병원"(특정 날짜·시간 불가)도 표현 불가.** 데모는 되나 실제 회의 하나를 못 잡음.
- **Hypothesis (검증됨)**: 연구 논지(공정 배분+익명 선호)를 시연하느라 그 밑에 깔려야 할 **기초 프리미티브 — 실제 날짜 범위 위에서 각자 '안 되는 시간'을 칠하는 것 — 을 통째로 건너뜀.** 논지는 있는데 논지를 얹을 바닥이 없어 실사용 0.
- **Weakness (자기반박)**: 순진하게 "날짜+가용성 그리드"만 넣으면 곧 **When2Meet/Doodle 클론**이라 차별점(히트맵 눈치우기 없이 시스템이 공정하게 하나 결정)을 잃음. → **결론: 바닥(범위+그리드)을 진짜로 깔되, 그 위에 '하나를 정해줌+익명 공정' 레이어는 유지.**
- **Decision (구현)**:
  ① **범위 선택**: 조직자가 생성 시 **이번 주/다음 주/이번 달**을 고르면 `lib/dates.ts`가 실제 평일 날짜 목록을 생성(로컬 TZ, 오늘 이후만). `Meeting`에 `scope/dates[]/hourStart/hourEnd` 추가.
  ② **슬롯을 실제 날짜로**: `Slot`을 `{id:`${date}-${hour}`, date, hour}`로 바꿈. 라벨은 `07/13(월) 오전 11시`. 엔진의 랭킹(Floor→아쉬움↓→미응답↓→선택↑→빠른날)은 N·슬롯 무관이라 정렬 키만 날짜 문자열로 교체.
  ③ **참여자 '안 되는 시간' 그리드**: `onboard` 화면을 추상 토글에서 **날짜×시간 그리드로 교체** — 칸을 탭해 `안 돼요(hard)`/`되도록 피해요(soft)` 2단계로 칠함, 날짜 헤더 탭 시 하루 전체. 데이터는 `busyHard[]/busySoft[]`(슬롯 id). soft는 여전히 익명 집계로만 사용해 차별점 유지.
  ④ 모델 변경 전파: `Participant.soft/hardDays` → `busyHard/busySoft`; preferences 라우트는 범위 밖 id 필터+hard우선; seed는 실제 날짜 슬롯에 페르소나별로 칠함; serialize가 `rangeLabel`·범위 노출.
- **Evidence**: `tsc --noEmit`+`next build` 통과. dev 서버 API E2E — 다음주(7/13~17) 범위 생성→'나' Mon 9-10 불가·Tue13 소프트 저장→seed 2명→recommend가 **07/13(월) 오전 11시** 추천, 김PO의 Mon 9-10 하드블록은 blocked로 정확히 분리, 박개발 미응답은 unknownRequiredNames로 표기. 라벨·rangeLabel(07/13~07/17) 정상.
- **Weakness**: (1) 그리드가 **탭-토글 방식** — 드래그 페인팅 미구현(월간 ~22일×9시=많은 칸을 하나씩 탭은 번거로움; 하루 전체 토글로 일부 완화). (2) 시간대 09~17 고정, 15/30분 단위·주말·업무외 시간 미지원. (3) 월간 범위는 가로 스크롤이 길어짐(폰 UX 검증 필요). (4) 소요시간 1시간 고정 — 연속 슬롯 병합(2시간 회의) 로직 없음. (5) 클릭 단위 브라우저 스모크는 미실행(API E2E로 대체).
- **Next Research**: (a) 드래그 페인팅(포인터+elementFromPoint, touch-action 충돌 해소)로 다칸 선택 속도 개선. (b) 월간 그리드 폰 사용성 — 주 단위 접기/히트맵 밀도. (c) 소요시간>1시간일 때 연속 가용 슬롯 탐색. (d) 제출 Q2·Q3의 해결 서사를 "안 되는 시간 칠하기(빼기식 입력)+범위 선택"을 1차 프리미티브로, 익명 공정 배분을 2차 레이어로 재배치.

## Entry 038 — 그리드 드래그 페인팅 (탭→쓸어서 칠하기) (2026-07-06)
- **Fact**: Entry 037 Next Research (a) 실행. 참여자 그리드를 칸별 `onClick` 토글에서 **포인터 드래그 페인팅**으로 교체 — 마우스·터치 공용(Pointer Events). 셀은 `<div data-slot>`, `.grid` 컨테이너가 `onPointerDown/Move/Up`을 받아 `document.elementFromPoint`로 실제 셀을 찾고 `setPointerCapture`로 드래그를 잡는다.
- **Decision (인터랙션 규칙)**: (1) **모드 결정** — 드래그 시작 칸이 이미 현재 레벨이면 `remove`(지우기), 아니면 `add`(칠하기). 시작점이 의도를 정한다(When2Meet식). (2) **지우기는 같은 레벨만** — hard 지우다가 옆의 soft 칸을 실수로 날리지 않게 `remove`는 현재 레벨과 같은 칸만 지운다. (3) **스크롤 vs 페인팅 충돌** — `.gcell`에만 `touch-action:none`을 줘서 셀에서 시작한 드래그는 페인팅, 헤더·시간칸·여백에서 시작하면 스크롤에 양보. 헤더 탭은 그대로 하루 전체 토글.
- **Evidence**: `tsc --noEmit`+`next build` 통과. (브라우저 클릭·드래그 스모크는 **claude-in-chrome 확장 미연결로 미실행** — 시각 확인 못 함. 드래그 로직은 표준 pointer-capture+elementFromPoint 패턴이나 실기기 검증은 남음.)
- **Weakness**: (1) 실브라우저/터치 실측 미완 — pointercapture+elementFromPoint 조합이 특정 브라우저·모바일 사파리에서 다르게 동작할 여지. (2) `touch-action:none`이라 셀 위에서 시작한 세로 스크롤은 페인팅으로 먹힘 — 긴 그리드에선 여백/헤더로만 스크롤. (3) 드래그 실행취소(undo) 없음 — 잘못 쓸면 다시 쓸어 지워야 함.
- **Next Research**: (a) 확장 연결되면 실제 클릭·드래그 스모크 + 모바일 터치 확인. (b) Entry 037의 나머지: 월간 그리드 폰 밀도, 소요시간>1시간 연속 슬롯, 15/30분 단위. (c) 잘못 칠했을 때 되돌리기(마지막 드래그 undo) 필요성 검토.

## Entry 039 — 가변 소요시간: 연속 블록 후보 + 블록 단위 상태 판정 (2026-07-06)
- **Fact**: Entry 037 Next Research (c) 실행. 회의 길이 고정(1h) → **1/2/3시간 가변**. `Meeting.durationHours` 추가. 후보를 "그 날 연속 D시간이 다 들어가는 시작 시각"으로 생성(`candidateStarts`: `h + dur <= hourEnd`). 조직자 생성 화면에 소요시간 세그먼트 추가.
- **Decision (블록 판정 규칙)**: 개인 상태를 시간 단위가 아니라 **블록 전체로 판정** — `personBlockState`: 블록 `[start, start+dur)` 안 **한 시간이라도 hard면 그 사람은 불가**, 하나라도 soft면 아쉬움, 미등록은 미응답. 끝시각 **exclusive**(D=2, 10시 시작 → 10·11시 점유, 12시는 안 건드림). 랭킹 로직(Floor→아쉬움↓→미응답↓→선택↑→빠른날)은 그대로, 평가 함수만 블록화.
- **Decision (정수 시간으로 한정)**: 30분/1.5h는 그리드를 30분 단위로 쪼개야 해 입력 UI가 커진다 → 이번엔 **정수 시간(1~범위시간수)만** 지원. 참여자 입력(시간 단위 칠하기)은 그대로 두고 후보 생성/평가만 블록화해 변경 최소화.
- **Evidence**: `tsc`+`next build` 통과. dev E2E — 2시간·다음주 생성 후 김PO가 **월 12시** hard 지정 → 추천에서 12시를 포함하는 블록(11–13, 12–14)은 **정확히 제외**, 10–12시(10·11시 점유)·9–11시·오후 블록은 유지. 라벨은 "07/13(월) 오전 9시~오전 11시"로 범위 표기. durationHours 왕복 저장 확인.
- **Weakness**: (1) **30분/1.5시간 미지원** — 실제 스탠드업·1:1은 30분이 흔한데 그리드가 시간 단위라 못 나눔(가장 큰 남은 갭). (2) 근무시간 09~18 고정. (3) 소요시간이 범위 시간 수를 넘으면 후보 0(서버가 clamp하지만 UX 안내는 없음). (4) 실브라우저 스모크는 확장 미연결로 여전히 미실행.
- **Next Research**: (a) 30분 단위 지원 여부 판단 — 그리드 행을 30분으로 쪼갤지, 아니면 "시작 시각만 30분 눈금"으로 절충할지. (b) 근무시간대(예: 오전만/야간) 커스터마이즈. (c) 확장 연결 후 create→grid→recommend 클릭 플로우 시각 검증.

## Entry 040 — 30분 눈금 지원: 시간의 원자 단위를 '분'으로 일반화 (2026-07-07)
- **Fact**: Entry 039 Next Research (a) 실행. 스탠드업·1:1 등 30분 회의가 흔한데 그리드가 시간 단위라 표현 불가했던 갭 해소. 시간의 원자 단위를 **정시(hour) → 자정부터 분(min)**으로 일반화. `Slot.hour` 제거·`Slot.min` 도입, id = `${date}-${min}`(예 09:30 → `-570`). `Meeting`에 `stepMin`(눈금, 30 고정)·`durationMin`(30/60/90/120) 도입, `durationHours` 제거.
- **Decision (일반화)**: `makeSlots`/`candidateStarts`/`personBlockState`를 전부 분·step 루프로 재작성. 블록 판정은 `for m=start; m<start+dur; m+=step`로 동일 로직. 라벨 `clock(min)`: 정시 "오전 9시", 반시간 "오전 9시 30분", 블록은 "9시~10시" 범위. **눈금은 30분 고정**(60분도 허용은 하되 UI는 30분) — 15분까지 열면 칸 수·정밀도 과잉이라 실사용 균형점으로 30분 선택.
- **Decision (입력 UI)**: 참여자 그리드 행을 30분으로 2배(9~18시 → 18행). 정시 행은 진하게, 30분 행은 옅은 라벨(`9:30`)+옅은 셀 배경으로 정시 경계가 눈에 들어오게. 드래그 페인팅(Entry 038)은 id만 분 기반으로 바뀌어 그대로 동작.
- **Evidence**: `tsc`+`build` 통과. dev E2E 2건 — (1) **30분 회의**: 김PO가 09:00만 hard → 09:00 blocked, **09:30/10:00/10:30 생존**(30분 해상도 증명). (2) **90분 회의**: 김PO가 10:00 hard → 10:00 포함 블록(9:00–10:30/9:30–11:00/10:00–11:30) 전부 제외, **10:30~12:00만** 추천(다중 스텝 블록 평가 증명). 라벨 "오전 10시 30분~낮 12시" 정상.
- **Weakness**: (1) 15분 단위·비정렬 시작(9:45) 미지원. (2) 근무시간 09~18 정시 경계 고정 — 커스터마이즈 없음. (3) 월간×30분이면 22일×18행=396칸 — 렌더/스크롤 부담, 폰 밀도 미검증(Entry 037 Weakness 잔존). (4) 실브라우저 스모크 여전히 미실행(확장 미연결).
- **Next Research**: (a) 월간+30분 그리드의 폰 사용성(주 접기/가상 스크롤/밀도) — 이제 칸 수가 커져 우선순위 상승. (b) 근무시간대 커스터마이즈(오전만/야간). (c) 확장 연결 후 create→30분 그리드 드래그→recommend 시각·터치 검증.

## Entry 041 — 월간 그리드 폰 사용성: 주 단위 페이지네이션 (2026-07-07)
- **Fact (문제)**: 월간×30분 = 22평일×18행 ≈ 396칸. 평평한 22열이 폰(390px)에서 가로로 길게 스크롤돼 (1) 주 맥락 상실, (2) "몇 주차 며칠" 위치 감각 없음, (3) 가로·세로 동시 스크롤로 길 잃음.
- **Hypothesis**: 근본 원인은 밀도가 아니라 **평평한 22열 구조**. 사람의 월간 멘탈모델은 '주'로 묶임. → 주 단위 페이지네이션으로 한 번에 한 주(≤5열)만 폭에 꽉 채우면 가로 스크롤 소멸 + 열 확대(~61px) + 맥락 회복.
- **Weakness (자기반박)**: "월간을 30분 단위로 칠하는 게 애초에 비현실적" — 맞음. 그러나 출력은 정확한 시각이어야 하므로(회의는 특정 시각에 열림) 모델은 30분 유지, 대신 입력 부담은 '하루 전체 탭'(기존) 같은 coarse 액션으로 낮춘다. **모델은 정밀하게, 입력은 거칠게.**
- **Decision**: `dates.groupByWeek(dates)`(월요일 기준)로 주 묶기. `weekIdx` 상태로 한 주만 렌더, `weeks>1`일 때만 `‹ N/M주차·날짜범위 ›` 네비 노출. 그리드는 `minmax(0,1fr)`로 폭 채움(`.grid.paged`), 가로 스크롤 제거. 하단 힌트에 '이 주 n칸 · 전체 N칸'. **엔진·데이터 모델 무변경** — 순수 렌더/네비 레이어.
- **Evidence**: `tsc`+`build` 통과. 날짜 로직 독립 검증 — thisMonth(오늘 7/7) 19평일 → **4주 분할**: 1/4(7/7~10, 4일)·2/4(7/13~17)·3/4(7/20~24)·4/4(7/27~31). 각 주 ≤5열. (주간 스코프는 weeks=1이라 네비 없이 그대로.)
- **Weakness**: (1) 여전히 한 주 안 30분×5일=90칸 세로 스크롤은 존재(허용 범위). (2) 주 간 이동 시 칠한 표시가 즉시 안 보여 '전체 n칸' 텍스트로만 안내 — 주별 요약 배지 없음. (3) 오전/오후 반나절 coarse 액션 미구현(하루 전체 탭만). (4) 실브라우저·터치 스모크 미실행(확장 미연결).
- **Next Research**: (a) 주별 '칠한 칸 있음' 인디케이터(네비 점/배지). (b) 오전(9-12)/오후(12-18) 반나절 1탭 차단. (c) 근무시간대 커스터마이즈. (d) 확장 연결 후 월간 페이지네이션 스와이프·드래그 시각 검증.

## Entry 042 — 초대코드 유실 방지: 조용한 폴백을 '조직자 경고'로 (2026-07-07)
- **Fact (문제)**: 배포(Vercel)에서 "초대 코드로 접속 시 '그런 초대 코드를 찾지 못했어요'(404)" 재현. 원인은 join 로직 버그가 아니라 **저장소 휘발** — KV(Upstash) 미연동 시 `store.ts`가 인메모리 Map으로 **조용히 폴백**하는데, 서버리스는 요청마다 인스턴스가 달라 조직자가 만든 코드가 참여자 인스턴스에 없다. 로컬 단일 프로세스에선 안 드러나 배포 후에야 터지는 함정.
- **Hypothesis**: 근본 문제는 폴백 자체가 아니라 **폴백이 조용하다는 것**. 조직자는 코드가 유실될 상태인지 모른 채 공유 → 참여자가 404를 맞고서야 발견. → 실패를 **공유 전에·조직자에게** 노출(fail-loud)하면 사고 자체를 예방한다.
- **Decision**: `GET /api/health`가 `{kv, serverless}` 반환(`usingKv()`+`process.env.VERCEL`). 클라이언트는 마운트 시 1회 조회해, 조직자 초대 화면 코드 토큰 바로 아래에 배너 노출. **심각도 2단계** — (a) `serverless && !kv`: 빨강 "다른 기기에서 접속하면 찾지 못할 수 있어요, 공유 전 KV 연동 필요"(실제 위험), (b) 로컬 `!kv`: 노랑 "로컬 메모리 모드, 재시작 시 코드 사라짐"(정보성). KV 정상이면 배너 없음. 위치를 **공유 버튼 위**에 둬 잘못된 공유 직전에 차단.
- **Decision (부수)**: 설정 화면 참여자 예시 6명→**3명(필수/필수/선택)**으로 축소 — 예시 과다가 인지부하만 키우고 '필수/선택' 개념 전달엔 최소 대비(필수2+선택1)면 충분. `MIN_PARTICIPANTS=2`라 삭제 동작도 유지.
- **Evidence**: `tsc --noEmit` 통과. `/api/health` 로컬 응답 `{"kv":false,"serverless":false}` 확인 → 로컬에선 노랑 배너 경로. 배포 시 `serverless:true`로 빨강 배너 전환됨(코드 경로 확정, 실배포 스모크는 미실행). create→join API는 단일 프로세스에서 200/201 정상(로직 무결 재확인).
- **Weakness**: (1) 실제 Vercel 배포에서 빨강 배너·연동 후 소멸을 **눈으로 검증 못 함**(확장이 로컬/배포 URL 접근 불가). (2) 배너는 조직자만 봄 — 이미 공유된 코드로 들어온 참여자의 404 문구엔 원인 설명 없음. (3) `process.env.VERCEL` 의존 — 타 호스팅(Netlify 등 서버리스)에선 `serverless:false`로 잡혀 노랑(과소경고)될 수 있음. (4) Upstash 실연동은 계정·OAuth 필요라 미완(코드·`.env.local` 템플릿만 준비).
- **Next Research**: (a) Vercel 배포+Upstash 연동 후 배너 2단계 실검증. (b) 참여자 404 화면에도 "조직자에게 재발급 요청" 안내 추가 검토. (c) `serverless` 감지를 `VERCEL||NETLIFY||...` 또는 "kv 없음=항상 경고"로 넓힐지 판단.

## Entry 043 — 주말 포함 토글 · 참여는 코드 대신 공유 링크로 (2026-07-07)
- **Fact (문제)**: (a) 후보 날짜가 평일(월~금)로 **하드코딩**(`weekdaysOfWeek`/`weekdaysOfMonth`)이라 주말 회의(스터디·동호회)를 못 잡음. (b) 조직자에게 **6자리 코드만** 줘서, 참여자는 앱 열기→코드 수동 입력→이름의 마찰을 겪음. (c) join 시 로스터에 없는 이름이면 새 참여자로 합류 → "아무 이름이나 입장" 됨.
- **Hypothesis**: (a) 주말 여부는 회의 성격에 따라 갈리므로 **조직자 선택지**로 노출(기본 평일). (b) 링크는 탭 1회로 코드 자동입력까지 끝내 마찰을 없앤다. 단 코드를 **없애지 않고** 링크를 1순위·코드를 폴백으로(구두·타 채널 전달엔 코드가 강함). (c)는 별도 결정 필요(아래 Weakness).
- **Decision (주말)**: `daysOfWeek/daysOfMonth(base, includeWeekend)`로 일반화(이름도 weekday→days로 정직하게), `buildRange(scope, today, includeWeekend=false)`. 생성 화면 스코프 아래 "평일만/주말 포함" 세그. 기본 평일(업무 회의 다수). 엔진·그리드·groupByWeek 무변경(mondayOf 기준이라 토·일도 같은 주로 묶임).
- **Decision (공유 링크)**: `?join=CODE` 딥링크 — 마운트 시 쿼리 파싱해 참여 화면으로 이동+코드 프리필(이름만 입력). 초대 화면 1순위 CTA "참여 링크 공유"(`navigator.share` → 실패/데스크톱은 clipboard 폴백), 2순위 "코드만 복사". 참여 화면 힌트도 링크 도착 시 "이름만 넣으면 돼요"로 분기.
- **Evidence**: `tsc` 통과. dates 단위검증 — nextWeek 평일 5일[월화수목금]/주말 7일[…토일], thisMonth 19일/25일. dev: 홈·`/?join=CODE`·`/api/health` 모두 200, 컴파일 에러 0, KV 연결 유지. (딥링크 실제 리다이렉트·share 시트는 클라JS라 curl 미검증 — 로직만 확인.)
- **Weakness**: (1) **참여 신원 모델 미결**(핵심): 현재 자유 입력이라 무단 입장·중복 신원("민수/김민수/김PO") 가능. 6명이 정해진 회의면 로스터에서 이름을 **고르게** 하는 게 맞는지 결정 필요 — 사용자에게 질의 중. (2) 딥링크·share 실브라우저/모바일 미검증(확장이 로컬 접근 불가). (3) 주말 포함 시 월간 그리드가 5열→7열로 늘어 폰 밀도(Entry 041) 재부담. (4) `?join` 코드 유효성 사전검증 없음 — 틀린 코드면 참여 시도 후에야 404.
- **Next Research**: (a) 참여 신원: 로스터-락(명단에서 선택) vs 오픈 vs 하이브리드 결정 후 join UX·API 개편. (b) 딥링크 도착 시 코드 유효성 프리체크(회의 제목·인원 미리 보여주기). (c) 주말 포함 월간 7열 폰 사용성 재점검.

## Entry 044 — 참여 신원: 자유 입력 → 로스터-락(명단에서 선택) (2026-07-07)
- **Fact (문제)**: 참여 시 이름을 아무거나 쳐도 입장됐다(테스트로 확인). 원인은 join이 로스터에 없는 이름을 **새 선택 참여자로 생성**하던 로직. 6명이 정해진 회의에서 (a) 무단 입장, (b) 같은 사람이 "민수/김민수/김PO"로 중복 신원 생성, (c) 조직자 집계 왜곡을 유발.
- **Decision (로스터-락)**: 조직자가 만들 때 넣은 **명단에 있는 이름만** 입장 허용. 사용자 선택으로 확정(vs 오픈/하이브리드). 근거 — 회의 성립 전제가 "정해진 N명"이고, 조직자는 이미 명단을 입력하므로 참여자는 **선택**만 하면 됨(입력 마찰↓ + 신원 무결성↑). 하이브리드('명단에 없어요' 추가)는 무단입장 여지를 남겨 이번엔 제외(추후 게스트 필요 시 재검토).
- **Decision (구현)**: 백엔드 `join`은 `participants.find(name===)` 매칭만, 없으면 **403 "명단에 없는 이름이에요"**(새 참여자 생성·저장 로직 삭제). 등록 여부와 무관하게 매칭 허용 → 링크로 재입장해 자기 시간 **수정** 가능. 클라이언트 참여 화면은 자유 텍스트 입력을 없애고, 코드 6자리 입력 시 `getMeeting`으로 로스터를 불러와 **이름 칩 선택**(제출자는 ✓, 다시 선택해 수정 가능). CTA는 `roster && 선택`일 때만 활성.
- **Evidence**: `tsc` 통과. dev API — 명단 이름(이팀장) join **200**, 임의 이름(홍길동) **403 "명단에 없는 이름이에요"**. 코드 6자리→로스터 fetch→칩 렌더 경로 구성. 딥링크(`?join`)로 들어오면 코드 프리필→자동 로스터 로드→이름만 선택.
- **Weakness**: (1) **이름 기반 신원의 한계** — 팀 내 동명이인이나 지인이 남의 이름 칩을 고르는 건 못 막음(인증 없음). 로스터-락은 '아는 이름들'로 범위만 좁힘. (2) 게스트(명단 밖 정당한 참여자) 유입 경로 없음 — 조직자가 미리 넣어야 함. (3) 칩 UX 실브라우저·모바일 터치 미검증(확장 로컬 접근 불가). (4) 로스터 이름이 닉네임('김PO')이면 참여자가 본인으로 못 알아볼 여지.
- **Next Research**: (a) 동명이인·오선택 대비 — 마지막 4자리 코드/이모지 등 경량 자기확인 검토. (b) 하이브리드 게스트 추가를 조직자 승인부 옵션으로 열지. (c) 로스터 이름 표기 가이드(실명 권장) 또는 조직자가 참여자에게 표시명 안내.

## Entry 045 — 코드 무한 발급 방지(세션당 1개·수정) + 근무시간대 조직자 선택 (2026-07-07)

- **Decision (근무시간대)**: 생성 화면에 시작/끝 시각 셀렉트(6–24시, `hourLabel`로 오전/오후/정오/자정). `dayStart<dayEnd` 강제(끝 옵션은 시작 초과만), 서버도 `hourEnd>hourStart` 아니면 400. range-note에서 '평일' 고정어 제거(주말 포함과 충돌).
- **Evidence**: `tsc` 통과. dev API — 생성(코드 MP2BED, 9~18) → 같은 코드 PUT(10~22, 나 제거·최PO 추가·이팀장 required 변경) 결과 **코드 동일·이팀장 id 승계 확인**. 끝≤시작(18~9) PUT → **400 "시간대 끝은 시작보다 늦어야 해요"**. (셀렉트·CTA 분기·업데이트 플로우는 클라JS라 브라우저 시각검증은 로컬 접근 제약으로 미실행.)
- **Weakness**: (1) 범위/시간대를 줄이는 수정 시, 이미 응답한 참여자의 busy 슬롯 id 일부가 새 그리드 밖이 돼 **조용히 무시**됨(그 사람이 더 가용한 것처럼 보임) — 공유 후 대폭 편집 시 정합성 리스크. (2) 업데이트에 낙관적 락 없음 — 조직자 편집과 참여자 응답이 겹치면 마지막 저장이 이김. (3) 시간대 셀렉트는 정시 단위(30분 시작 경계 불가). (4) 실브라우저·모바일 셀렉트 UX 미검증.
- **Next Research**: (a) 공유 후 편집은 별도 확인 모달/경고(참여자 응답 영향 고지)로 감쌀지. (b) decidedSlotId뿐 아니라 범위 밖 busy 슬롯도 정리(prune)해 저장할지. (c) 30분 시작 경계 시간대 지원 필요성.
